## 版本策略

### 为什么是 2.2.0 而不是 3.0.0？

虽然这次升级涉及重大的内部重构（从 raster/sp 迁移到 terra/sf），但我们选择次要版本更新（2.2.0）而不是主要版本更新（3.0.0），原因如下：

1. **向后兼容性**：
   - 所有现有用户代码无需修改即可工作
   - 旧函数名保留为废弃别名
   - 自动格式转换确保兼容性

2. **渐进式升级**：
   - 用户可以逐步迁移到新 API
   - 有充足的过渡期
   - 降低升级风险

3. **语义化版本规范**：
   - 主版本（3.0.0）：破坏性变更，不兼容
   - 次版本（2.2.0）：新功能，向后兼容
   - 补丁版本（2.1.1）：bug 修复

## 废弃策略

### 废弃时间表

| 版本 | 状态 | 说明 |
|-----|------|------|
| 2.2.0 | 标记废弃 | 旧函数可用，显示警告 |
| 2.3.0 | 部分移除 | 移除使用频率极低的函数 |
| 3.0.0 | 完全移除 | 所有废弃函数移除 |

### 废弃函数列表

#### 重命名函数（保留别名）
```r
# 旧名称 → 新名称
readmesh()        → read_mesh()
readriv()         → read_river()
readatt()         → read_att()
MeshData2Raster() → mesh_to_raster()
sp.RiverOrder()   → calc_river_order()
# ... 等等
```

#### 移除函数（推荐替代）
```r
# 移除函数 → 推荐替代
read.df()  → read.table() 或 readr::read_delim()
write.df() → write.table() 或 readr::write_delim()
count()    → table()
```

## 用户迁移指南

### 对于新用户
- 直接使用新 API（snake_case 函数名）
- 参考新文档和示例
- 享受更好的性能和错误提示

### 对于现有用户

#### 选项 1：继续使用旧 API（短期）
```r
# 现有代码无需修改
library(rSHUD)
mesh <- readmesh("model.mesh")  # 会显示废弃警告
```

**优点**：无需立即修改代码
**缺点**：会看到废弃警告，未来版本可能不可用

#### 选项 2：逐步迁移（推荐）
```r
# 新代码使用新 API
library(rSHUD)
mesh <- read_mesh("model.mesh")  # 新函数名

# 旧代码保持不变，逐步更新
```

**优点**：平滑过渡，降低风险
**缺点**：需要一些时间

#### 选项 3：一次性迁移（激进）
```r
# 使用查找替换更新所有函数名
# readmesh → read_mesh
# readriv → read_river
# 等等
```

**优点**：一劳永逸，代码现代化
**缺点**：需要测试所有代码

### 迁移工具

我们提供了迁移辅助工具：

```r
# 检查代码中使用的废弃函数
rSHUD::check_deprecated_usage("path/to/your/code.R")

# 自动重命名函数（生成新文件）
rSHUD::migrate_code("old_code.R", "new_code.R")
```

## 常见问题

### Q1: 我必须立即升级吗？
A: 不必须。v2.1.0 仍然可用。但 v2.2.0 提供更好的性能和功能。

### Q2: 升级会破坏我的代码吗？
A: 不会。v2.2.0 完全向后兼容，现有代码无需修改。

### Q3: 废弃警告很烦人，如何关闭？
A: 可以临时关闭：`options(rSHUD.suppress_deprecated = TRUE)`
   但建议迁移到新 API。

### Q4: 新旧函数有性能差异吗？
A: 内部实现相同，性能一致。只是函数名不同。

### Q5: 何时必须迁移到新 API？
A: v3.0.0 之前。预计至少有 1-2 年的过渡期。

### Q6: 如何获取迁移帮助？
A: 
- 查看迁移指南 vignette
- 查看 NEWS.md 文件
- 在 GitHub 提问
- 发送邮件给维护者
