# 需求文档

## 简介

本文档定义了 rSHUD R 包现代化升级的需求。rSHUD 包是 SHUD（Simulator for Hydrologic Unstructured Domains，非结构化域水文模拟器）建模系统的综合工具箱。本次现代化升级旨在更新包以使用当前最佳实践、现代空间数据库和改进的代码质量标准，同时为现有用户保持向后兼容性。

### 开发策略

现代化升级遵循最小工程化原则。实施采用渐进式方法，首先实现核心功能，后续可扩展到高级特性。设计结构具有可扩展性，同时保持初始实现的专注和可管理性。

### 开发原则

1. **避免重复开发**：复用现有函数和模块，不重复造轮子
2. **模块化设计**：每个模块职责单一，降低耦合度
3. **最小改动原则**：新功能扩展不应影响现有核心功能
4. **避免冗余**：一个功能只在一处实现，通过参数化支持不同场景
5. **单一职责**：每个函数只做一件事，便于测试和维护

## 术语表

- **rSHUD 包**：为 SHUD 水文模型系统提供工具的 R 包
- **传统空间库**：已废弃的 R 空间包（raster、sp、rgeos、rgdal）- rSHUD 不再支持
- **现代空间库**：当前的 R 空间包（terra、sf）- rSHUD 唯一支持的空间库
- **SHUD 模型**：非结构化域水文模拟器 - 一个水文模型
- **网格域**：用于空间离散化的非结构化三角网格
- **河流网络**：河流和溪流系统的空间表示
- **验证层**：参数验证和错误检查函数
- **核心函数**：主要工作流中使用的基本函数
- **废弃函数**：标记为将来移除但仍可用的函数（仅限函数名，不包括数据格式）

## 需求

### 需求 1：空间库迁移

**用户故事：** 作为包维护者，我希望从传统空间库（raster/sp/rgeos）完全迁移到现代库（terra/sf），以便包使用积极维护的依赖项并获得更好的性能。

#### 验收标准

1. 当包使用栅格操作时，rSHUD 包应当直接使用 terra 库函数
2. 当包使用矢量空间操作时，rSHUD 包应当直接使用 sf 库函数
3. 当用户提供传统空间对象作为输入时，rSHUD 包应当拒绝并提供清晰的错误消息指导用户使用 terra/sf
4. 对于大数据集操作，rSHUD 包应当实现至少 20% 的性能提升（相比传统实现）
5. 当 terra 或 sf 包不可用时，rSHUD 包应当在错误消息中提供清晰的安装说明

### 需求 2：函数命名向后兼容性

**用户故事：** 作为现有 rSHUD 用户，我希望函数重命名时有过渡期，以便我有时间更新代码。

#### 验收标准

1. 当函数名称已更改时，rSHUD 包应当提供旧函数名作为废弃别名
2. 当调用废弃函数时，rSHUD 包应当显示警告消息，指示新函数名
3. rSHUD 包应当在至少两个次要版本发布中维护废弃函数
4. 当用户使用传统空间对象（raster/sp）时，rSHUD 包应当拒绝并提供清晰的迁移指导
5. rSHUD 包应当在文档中明确说明不再支持 raster/sp 格式

### 需求 3：代码质量和标准

**用户故事：** 作为包开发者，我希望代码库遵循 R 包开发最佳实践，以便代码可维护且一致。

#### 验收标准

1. rSHUD 包应当对所有新函数和重命名函数使用 snake_case 命名约定
2. 当函数被导出时，rSHUD 包应当提供完整的 roxygen2 文档，包括参数、返回值和示例
3. rSHUD 包应当验证所有必需的函数参数，并在验证失败时提供清晰的错误消息
4. 当函数参数有约束时，rSHUD 包应当检查约束并用描述性消息报告违规
5. rSHUD 包应当通过 R CMD check，零错误零警告

### 需求 4：函数参数验证

**用户故事：** 作为包用户，我希望在提供无效输入时获得清晰的错误消息，以便快速识别和修复问题。

#### 验收标准

1. 当缺少必需参数时，rSHUD 包应当停止执行并显示消息说明需要哪个参数
2. 当参数类型不正确时，rSHUD 包应当停止执行并显示消息说明预期类型和接收类型
3. 当数值参数超出有效范围时，rSHUD 包应当停止执行并显示消息说明有效范围
4. 当空间对象具有不兼容的坐标系统时，rSHUD 包应当停止执行并显示消息描述不兼容性
5. 当参数值可能导致意外结果时，rSHUD 包应当显示警告消息

### 需求 5：核心 GIS 函数现代化

**用户故事：** 作为包用户，我希望核心 GIS 函数使用现代空间库，以便受益于改进的性能和功能。

#### 验收标准

1. rSHUD 包应当使用 terra 库实现 MeshData2Raster 函数
2. rSHUD 包应当使用 terra 和 sf 库实现 sp2raster 函数
3. rSHUD 包应当使用 sf 库实现河流处理函数
4. rSHUD 包应当实现网格生成函数以输出 sf 兼容格式
5. 当 GIS 函数接受空间输入时，rSHUD 包应当支持传统和现代空间对象类型

### 需求 6：函数命名标准化

**用户故事：** 作为包用户，我希望函数命名一致，以便轻松发现和记住函数名。

#### 验收标准

1. rSHUD 包应当对所有数据读取函数使用 read_ 前缀
2. rSHUD 包应当对所有数据写入函数使用 write_ 前缀
3. rSHUD 包应当对所有计算函数使用 calc_ 前缀
4. rSHUD 包应当对所有绘图函数使用 plot_ 前缀
5. 当函数名称更改时，rSHUD 包应当提供使用旧名称的废弃别名

### 需求 7：文档完整性

**用户故事：** 作为包用户，我希望所有函数都有完整的文档，以便正确理解如何使用它们。

#### 验收标准

1. 当函数被导出时，rSHUD 包应当在其文档中包含标题和描述
2. 当函数有参数时，rSHUD 包应当记录每个参数的类型、用途和有效值
3. 当函数返回值时，rSHUD 包应当记录返回值结构和类型
4. 当函数已迁移时，rSHUD 包应当在其文档中包含迁移说明
5. rSHUD 包应当为每个导出函数提供至少一个工作示例

### 需求 8：测试覆盖

**用户故事：** 作为包维护者，我希望有全面的测试，以便可以自信地进行更改而不破坏功能。

#### 验收标准

1. rSHUD 包应当为所有核心函数包含单元测试
2. rSHUD 包应当实现至少 70% 的核心功能测试覆盖率
3. 当发生空间格式转换时，rSHUD 包应当包含验证正确转换的测试
4. rSHUD 包应当包含错误处理和参数验证的测试
5. 当需要向后兼容性时，rSHUD 包应当包含使用传统输入格式的测试

### 需求 9：直接使用现代空间库

**用户故事：** 作为包开发者，我希望直接使用 terra 和 sf 的原生函数，以便代码简洁、可维护，用户学习标准 R 空间生态系统。

#### 验收标准

1. rSHUD 包应当直接调用 terra 函数进行栅格操作（如 terra::rast(), terra::crop()）
2. rSHUD 包应当直接调用 sf 函数进行矢量操作（如 sf::st_read(), sf::st_buffer()）
3. rSHUD 包不应当提供 terra/sf 函数的封装层
4. 当需要在 terra 和 sf 之间转换时，rSHUD 包应当直接使用 terra::vect() 或 sf::st_as_sf()
5. rSHUD 包应当在文档中引导用户学习 terra 和 sf 的官方文档

### 需求 10：依赖管理

**用户故事：** 作为包维护者，我希望正确管理依赖项，以便用户可以可靠地安装和使用包。

#### 验收标准

1. rSHUD 包应当在 DESCRIPTION 的 Imports 部分列出 terra (>= 1.7-0) 和 sf (>= 1.0-0)
2. rSHUD 包应当完全移除 raster、sp 和 rgeos 依赖（不在 Imports 也不在 Suggests）
3. rSHUD 包应当在 DESCRIPTION 中指定最低 R 版本为 4.0.0
4. rSHUD 包应当移除 LinkingTo 中的 sp 依赖
5. 当可选功能需要额外包时，rSHUD 包应当检查包可用性并提供有用的消息

### 需求 11：函数废弃和移除

**用户故事：** 作为包维护者，我希望有明确的废弃策略，以便用户有时间在函数被移除之前更新代码。

#### 验收标准

1. 当函数被废弃时，rSHUD 包应当使用 .Deprecated 函数调用标记它
2. 当调用废弃函数时，rSHUD 包应当显示替代函数名称
3. rSHUD 包应当在 NEWS.md 文件中记录所有废弃函数
4. rSHUD 包应当在至少两个次要版本发布中维护废弃函数
5. 当函数被移除时，rSHUD 包应当在 NEWS.md 中记录移除并提供迁移指导

### 需求 12：性能优化

**用户故事：** 作为包用户，我希望大数据集的性能得到改善，以便分析更快完成。

#### 验收标准

1. 当执行栅格操作时，rSHUD 包应当使用提供改进性能的 terra 函数
2. 当执行矢量操作时，rSHUD 包应当使用提供改进性能的 sf 函数
3. rSHUD 包应当在可能的情况下使用向量化操作而不是循环
4. rSHUD 包应当为大型结果对象预分配内存
5. 当存在性能关键操作时，rSHUD 包应当保持或改善执行时间（相比传统实现）

### 需求 13：迁移文档

**用户故事：** 作为升级到 2.2 版本的现有用户，我希望有清晰的迁移指导，以便高效更新代码。

#### 验收标准

1. rSHUD 包应当在包中提供迁移指南文档
2. 当函数名称已更改时，rSHUD 包应当在迁移指南中记录旧名称和新名称
3. 当空间对象类型已更改时，rSHUD 包应当在迁移指南中提供转换示例
4. rSHUD 包应当为常见工作流包含前后代码示例
5. rSHUD 包应当在 NEWS.md 文件中记录所有破坏性更改

### 需求 14：冗余函数清理

**用户故事：** 作为包维护者，我希望移除冗余和不完整的函数，以便包更易于维护和理解。

#### 验收标准

1. 当函数重复基础 R 功能时，rSHUD 包应当废弃它们，改用基础 R 函数
2. 当函数不完整或实验性时，rSHUD 包应当完成它们或移除它们
3. 当函数功能重叠时，rSHUD 包应当将它们合并为单个设计良好的函数
4. rSHUD 包应当在 NEWS.md 中记录所有移除的函数及推荐的替代方案
5. 当函数被移除时，如果用户尝试调用它们，rSHUD 包应当提供有用的错误消息

### 需求 15：主接口函数更新

**用户故事：** 作为包用户，我希望主接口函数只接受现代空间格式，以便获得最佳性能和清晰的 API。

#### 验收标准

1. 当调用 autoBuildModel 函数时，rSHUD 包应当只接受 terra 和 sf 格式的空间输入
2. 当用户提供传统格式（raster/sp）时，rSHUD 包应当拒绝并提供清晰的错误消息和迁移指导
3. 当 autoBuildModel 函数处理数据时，rSHUD 包应当直接使用 terra 和 sf 函数
4. rSHUD 包应当返回 terra 和 sf 格式的结果
5. rSHUD 包应当在文档中提供使用 terra/sf 的完整示例

### 需求 16：代码复用和模块化

**用户故事：** 作为包开发者，我希望代码库遵循模块化设计原则，以便代码可复用和可维护。

#### 验收标准

1. 当多个函数中执行类似操作时，rSHUD 包应当将公共逻辑提取到共享实用函数中
2. 当创建新函数时，rSHUD 包应当复用现有函数而不是重复代码
3. rSHUD 包应当将相关函数组织到具有清晰边界的逻辑模块中
4. 当函数有多个职责时，rSHUD 包应当将其拆分为更小的单一用途函数
5. 当重构函数时，rSHUD 包应当确保更改不影响不相关的功能

### 需求 17：渐进式实现

**用户故事：** 作为包维护者，我希望渐进式实现更改，以便项目在每个阶段都保持可管理和可测试。

#### 验收标准

1. rSHUD 包应当在添加高级功能之前首先实现核心功能
2. 当实现新功能时，rSHUD 包应当确保每个阶段都完全功能化并经过测试
3. rSHUD 包应当设计可扩展的接口以便未来增强
4. 当计划复杂功能时，rSHUD 包应当首先实现最小可行版本
5. 当阶段完成时，rSHUD 包应当在进入下一阶段之前验证其正确工作

### 需求 18：参数化而非重复

**用户故事：** 作为包开发者，我希望使用参数化来处理变化，以便避免代码重复。

#### 验收标准

1. 当函数具有类似逻辑但有细微变化时，rSHUD 包应当使用参数来控制行为
2. rSHUD 包应当避免为类似操作的每个变体创建单独的函数
3. 当函数支持多种模式时，rSHUD 包应当使用清晰的参数名称来控制模式
4. 当默认行为是常见的时，rSHUD 包应当提供合理的默认参数值
5. rSHUD 包应当记录所有控制函数行为变化的参数
