# rSHUD 升级总结（统一口径版）

## 文档目的

本文件用于统一 rSHUD 升级项目的最终口径，解决历史文档中“严格断兼容”和“过渡兼容”两套说法并存的问题，作为后续开发与发布的对齐依据。

## 升级背景

rSHUD 的升级核心目标是将空间处理从传统栈（`raster/sp/rgeos/rgdal`）迁移到现代栈（`terra/sf`），同时提升代码质量、文档完整性与测试可维护性。

## 最终策略（已对齐）

### 1) 内部实现策略

- 包内部核心空间逻辑以 `terra/sf` 为主。
- 新实现优先采用现代 API（向量化、统一参数校验、清晰错误消息）。

### 2) 外部 API 策略

- 用户侧以“平滑迁移”为原则：
  - 旧函数名可保留为过渡别名（`.Deprecated`）。
  - 新文档、示例、教程以 snake_case 新命名为主。
- 命名规范统一：`read_*`、`write_*`、`calc_*`、`plot_*`。

### 3) 兼容性策略

- 不再新增新的传统空间栈依赖能力。
- 对历史工作流采取“可迁移、可提示、可逐步替换”的过渡方式。
- 迁移指南与错误消息必须明确给出替代路径。

## 关键决策记录

1. 统一口径优先于新增功能：先消除文档与实现冲突，再推进功能扩展。
2. “内部现代化 + 外部过渡兼容”作为当前阶段主线。
3. demo 与文档应引导新 API，不再强化旧命名。
4. 避免重复实现：同名功能只保留单一权威实现，防止加载顺序导致行为漂移。

## 实施方式（阶段化）

升级采用分阶段实施，核心顺序如下：

1. 基础设施与验证层
2. GIS 核心迁移
3. 网格模块迁移
4. 河流模块迁移
5. I/O 模块更新
6. 可视化模块更新
7. 主接口更新
8. 清理与标准化
9. 文档与测试补齐
10. 最终验证与发布准备

详细任务拆解见 `.kiro/specs/rshud-modernization/tasks.md`。

## 当前状态总结

### 已完成（总体）

- 现代化升级方案、设计与任务清单已建立并执行。
- 大量模块已完成重构与文档化。
- 命名标准化（snake_case）和参数校验体系已推进。
- 迁移文档、版本说明、发布清单等配套文档已形成。

### 仍需持续收敛

- 文档口径统一：移除互相冲突的表述（如“完全拒绝旧输入”与“过渡兼容”并存）。
- 环境一致性验证：确保测试环境依赖版本满足要求（尤其 `terra`/`sf`）。
- demo 端到端可运行性：以当前主分支实际导出函数与行为为准进行验证。

## 与早期总结的主要差异

本文件替代早期“v3.0 主线 + compat 层中心化”的描述，改为当前发布线上的统一决策版本。主要变化：

- 时间线从早期规划口径切换为当前发布节奏（`v2.2.x` 阶段）。
- 兼容策略由“单一极端口径”改为“分层策略”（内部现代化、外部过渡兼容）。
- 强调“文档与实现一致”作为优先级。

## 发布与迁移建议

### 对用户

- 新项目：直接使用 snake_case 新函数名。
- 旧项目：可逐步迁移，优先替换高频核心函数。
- 遇到废弃警告：按迁移指南替换，不建议长期忽略。

### 对开发者

- 新增功能不得引入新的传统空间栈路径。
- 涉及接口调整时，必须同步更新 `NEWS.md`、README、迁移文档与 demo。
- 每次发布前，执行“文档一致性检查 + demo 可运行检查 + check/test 检查”。

## 结论

rSHUD 升级已从规划阶段进入收敛阶段。当前工作的核心不再是继续扩大改动范围，而是：

1. 统一口径
2. 固化决策
3. 清理冲突
4. 确认可运行性

在此基础上再推进后续版本迭代，才能保证升级成果稳定、可维护、可交付。
